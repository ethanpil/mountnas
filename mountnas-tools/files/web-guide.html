Philosophy ↑

MountNAS is a diskless Alpine Linux NAS that runs entirely from RAM off a USB stick. The stick holds the OS and your saved configuration; your data lives on ordinary mounted drives. It is built for power users who want a small, transparent, hands-on system — every moving part is a readable shell script you can inspect (and, in a pinch, patch) on the box.

  • The OS is rebuilt in RAM at every boot from packages on the stick plus a small config overlay. A reboot is a factory-fresh OS with your config applied.
  • Nothing persists until you commit. nas commit saves your /etc changes back to the stick. This is a feature: experiments are free, and a reboot always recovers a clean system.
  • Storage is plain Linux. Disks are ordinary filesystems declared in /etc/fstab — no proprietary volume format. Any Linux machine can read your data disks.
  • Data services are supervised. Docker, Samba and NFS are held until the data disk is confirmed mounted, so a missing disk can never quietly fill RAM.
  • CLI-first, appliance discipline. One tool (nas) drives everything; optional extras (web dashboard, notifications, VPNs, UPS) are off until you enable them and safe to ignore.

How it works ↑

Boot sequence

  1. Firmware boots the USB stick (BIOS or UEFI); the kernel + initramfs load.
  2. Your saved config overlay (from the MNASCFG partition) is applied.
  3. The base system + any packages you added install into a RAM filesystem from the on-stick package snapshot and /cfg/cache — no network needed.
  4. The mountnas supervisor mounts your fstab disks, then starts Docker/Samba/NFS once /mnt/nasdata is up.

What lives where

PlaceWhatSurvives reboot?
RAM (/)The running OS, all live changesNo — until nas commit
USB BOOTKernel, packages, boot loadersYes (replaced by nas upgrade)
USB MNASCFG (/cfg)Committed config overlays, package cache, operations logYes
Data disks (/mnt/*)Your files, Docker data, appdata, backupsYes — never touched by commits or upgrades
The one rule that matters: changes live in RAM until nas commit. That includes passwords, fstab edits, installed packages, and Samba users. nas status and the shell prompt track the unsaved count for you.

First boot ↑

  1. Write the release image to a USB stick (4 GB+) with Etcher or dd, boot from it.
  2. Log in as root (no password). The setup wizard starts automatically: hostname, root password, timezone, network — then it saves.
  3. Add your disks (next section), then nas commit.

Finding a headless box

  • The console login screen shows the IP and hostname.local before login — a monitor is enough.
  • mDNS is on: ssh root@mountnas.local works on most LANs.
  • Pre-seed an SSH key: after flashing, drop a file named authorized_keys onto the FAT BOOT partition — it is installed at every boot.
  • The fresh image permits passwordless root SSH so you can get in; the wizard disables that automatically once a password is set.

Storage ↑

Disks are managed in /etc/fstab — like any Linux box. Mount everything under /mnt. One mountpoint is special: /mnt/nasdata must exist to hold Docker data, appdata and backups; Docker/Samba/NFS stay off until it is mounted.

Add a disk

nas disks                         # find the disk; copy its paste-ready fstab line
mkfs.ext4 -L nasdata /dev/sdX1    # blank disk? format it first
vi /etc/fstab                     # e.g.: UUID=<uuid>  /mnt/nasdata  ext4  rw,noatime,nofail  0 2
nas status                        # pre-flight check of your fstab
nas restart                       # mount + start services, no reboot
nas commit                        # persist it
  • Always include nofail — a missing disk must never hang the boot.
  • Mountpoints are created for you from fstab at each boot — no mkdir needed (and do not use x-mount.mkdir).
  • Network filesystems (NFS/CIFS) are never mounted at boot and are refused as /mnt/nasdata — a dead remote must not stall the box. Mount them manually or use them inside Docker.
  • nasdata on a shared disk? Use a bind mount: /mnt/disk1/nasdata  /mnt/nasdata  none  bind,nofail  0 0 (real disk listed first).

Pool disks into one mount (mergerfs)

/mnt/disk1:/mnt/disk2  /mnt/pool  fuse.mergerfs  nofail,allow_other,use_ino,category.create=mfs  0 0

Add after the member disks in fstab; nas commit and reboot (the pool is mounted by the boot path). Keep /mnt/nasdata out of the pool.

Parity (SnapRAID)

  1. Mount data + parity disks in fstab like any other disk.
  2. Configure /etc/snapraid.conf (a commented template ships). Keep /mnt/nasdata out of the array.
  3. Schedule sync/scrub and pipe the result into a notification:
crontab -e
0 3 * * *  snapraid sync 2>&1 | nas notify "snapraid sync"
0 5 * * 0  snapraid scrub 2>&1 | nas notify "snapraid scrub"

SnapRAID gives parity (recover a failed disk), mergerfs gives one namespace — they compose well. Then nas commit.

Sharing & services ↑

Samba (Windows/macOS shares)

vi /etc/samba/smb.conf            # add a [share] block (template included)
adduser -D alice                  # system user (if new)
smbpasswd -a alice                # samba password
rc-service samba restart
nas commit

NFS

vi /etc/exports
rc-service nfs restart && nas commit

Docker

Runs with data-root = /mnt/nasdata/docker. Keep compose files and appdata under /mnt/nasdata so they survive a dead USB and travel with the data. docker compose is included.

Remote access (all off by default)

  • Tailscale: rc-update add tailscale default && rc-service tailscale start && tailscale up && nas commit
  • ZeroTier: rc-update add zerotier-one default && rc-service zerotier-one start && zerotier-cli join <net> && nas commit
  • WireGuard: plain kernel WireGuard, wg/wg-quick included.

UPS (NUT)

NUT is baked in, off by default. Determine your UPS USB parameters, configure /etc/nut/ (nut.conf, ups.conf, upsd.conf, upsmon.conf, with SHUTDOWNCMD "/sbin/poweroff"), then rc-update add nut-upsd default && rc-update add nut-upsmon default && nas commit. A power cut discards uncommitted changes — commit early and often either way.

Admin users

adduser bob wheel && nas commit    # wheel = doas (sudo) rights

Monitoring & alerts ↑

nas status — the pre-flight check

nas status validates health and storage config without waking sleeping disks; exit code 1 on any failure makes it cron/monitoring friendly. --deep adds SMART/SnapRAID/time-sync; --json is machine-readable for Uptime Kuma, Zabbix, Homepage, etc.

This web dashboard

Enable/disable with nas web on [port] / nas web off (default port 8080, then nas commit). It is deliberately read-only: a root-run job renders static pages into RAM every 2 minutes and a tiny server (running as nobody) serves them — no request-time code, no forms, nothing to attack, no disk writes. Manage the box over SSH. /system.html adds hardware/CPU detail, your added packages, and the last 100 syslog lines; /status.json serves the dashboard data machine-readable.

Notifications

List sinks in /etc/mountnas/notify.conf, one per line; every alert goes to all of them:

ntfy:https://ntfy.sh/your-secret-topic          # push to your phone, no relay needed
email:you@example.com                           # needs /etc/msmtprc (msmtp) configured
webhook:https://example.com/hook                # generic JSON POST
slack:https://hooks.slack.com/services/...      # also Mattermost-compatible
discord:https://discord.com/api/webhooks/...
gotify:https://gotify.example.com/message?token=...

Test with nas notify --test, persist with nas commit. What alerts automatically: data-disk loss (disconnect, dead mount, filesystem gone read-only — once, on the transition), failed upgrades, and anything you pipe in yourself (... | nas notify "subject"). Sink docs: ntfy, msmtp (email), gotify.

SMART disk health (smartmontools)

smartd runs out of the box and never wakes spun-down disks. For failure alerts, add ONE line to /etc/smartd.conf and restart smartd:

DEVICESCAN -n standby,q -m root -M exec /usr/libexec/mountnas/smartd-notify   # via your sinks
DEVICESCAN -n standby,q -m you@example.com                                    # or classic email

Health digest (optional)

crontab -e
0 8 * * 1  /usr/libexec/mountnas/health-digest    # Monday-morning summary via your sinks

Operations log

nas history shows every setup, commit (with note), rollback, backup, upgrade and shutdown — timestamp, who, outcome. Zero setup; persists without a commit.

Persistent system logs (optional)

nas logs --persist on moves syslog to /mnt/nasdata/logs (rotated automatically) so a crash leaves history behind — tradeoff: periodic writes keep that disk awake.

Maintenance ↑

Committing & rolling back config

nas changes --diff                 # review what would be saved
nas commit -m "added media share"  # save, with a label
nas rollback --list                # previous saved configs (with your labels)
nas rollback 1 && nas reboot       # restore one (applies at next boot)

Backup (the rollback net)

nas backup images the whole boot USB (OS + saved config) to a verified gzip file — default /mnt/nasdata/backups. Copy it off the box. Recovery = write it to a different stick and boot. Data disks are separate — back those up with restic/rclone (both included).

Upgrading

nas upgrade --check                                  # is a newer release out?
nas backup                                           # required first — no automatic rollback
nas upgrade https://github.com/.../mountnas-X.img.gz # or a local file; then reboot

Upgrades rewrite the OS in place; your config and data are untouched, added packages are preserved. See UPGRADE.md in the repo for details and recovery.

Installing packages

apk add <pkg> works out of the box (repos are pinned to this release). After installing: nas commit — packages are cached on the stick and reinstall at every boot, even offline. Missing device firmware: find the file with dmesg | grep -i firmware, apk add linux-firmware-<vendor>, commit, reboot.

Recovery

  • Dead USB stick: write your latest nas backup image to a new stick and boot — OS and config restored exactly. No backup? Flash a fresh release and reconfigure; data disks are untouched.
  • Lost root password: mount MNASCFG on another Linux box, unpack the overlay, blank the root hash in etc/shadow, repack, boot, passwd, nas commit. (Full steps in the README.)
Never re-flash the release image over a running box (it erases the saved config — the image is for first install; upgrades go through nas upgrade), and never boot with two MountNAS sticks attached (their disk labels collide).

Command manual ↑

Run as nas <command>. Every command has focused help: nas <command> --help. Tab completion ships for bash and zsh.

CommandDoes
setupGuided first-run wizard (hostname, password, timezone, network). Auto-runs at first login; safe to re-run.
status [--deep|--json]Health + storage-config checks; exit 1 on failure. --deep adds SMART/SnapRAID (may wake disks); --json for monitoring.
disks [--json]Every disk: identity, partitions, temps (spin-down safe), fstab state, paste-ready fstab lines.
restartRe-mount fstab disks and (re)start Docker/Samba/NFS — no reboot. Run after editing fstab.
changes [--diff]What commit would save; --diff shows the content. (alias: changed)
commit [-m "note"]Save in-RAM config to the stick. Label it — labels show in rollback --list. (alias: save)
rollback [--list|<n>]Config time machine: restore a previous committed overlay (applies at next boot; roll-forward possible).
backup [--to <dir|file>]Verified image of the whole boot USB. Copy it off-box; it is the upgrade/dead-stick recovery net.
upgrade [--check|--yes] <img|url>In-place OS upgrade from a release image; config/data untouched. Requires a backup first.
history [-n N|--all]Append-only operations log: setups, commits, rollbacks, backups, upgrades, power events — with who and when.
notify [--test|<subject> [body]]List/test notification sinks, or send an ad-hoc message (body pipeable) — cron friendly.
web [on [port]|off|status]This read-only dashboard + guide. Off by default; default port 8080.
logs [-n N|-f|--persist ...]View syslog; optionally persist it to the data disk with rotation.
reportSecrets-free diagnostics bundle in /tmp for bug reports.
shutdown / reboot [--yes|--save]Power off / reboot with an unsaved-changes gate; --save commits first.
version / about / helpRelease info, project info, and the two-page command overview.

Troubleshooting ↑

SymptomFix
Docker/Samba/NFS will not startnas status — almost always the data disk is not mounted. Fix fstab, re-check, nas restart.
Settings gone after rebootThey were never committed. Check /cfg is mounted (nas status), then nas commit.
Cannot find the boxTry mountnas.local; or attach a monitor — the IP is shown above the login prompt.
Not reachable on the networkWired DHCP is auto-configured on first boot; check cable/link. Custom config: /etc/network/interfaces, rc-service networking restart, commit.
Custom init.d service vanishedlbu does not track /etc/init.d. Once: lbu include /etc/init.d/<name>, then commit.
commit fails: "tar: empty archive"RAM root is full (df -h /). Free space or reboot, then commit.
Dashboard not loadingnas web status; check the port (default 8080) and that you committed after nas web on.
Notifications not arrivingnas notify --test, then tail /var/log/mountnas.log. Email sinks additionally need /etc/msmtprc.

Deeper diagnosis: nas report writes a secrets-free bundle for a bug report.
