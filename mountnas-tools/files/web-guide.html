Philosophy ↑

MountNAS is a diskless Alpine Linux NAS that runs entirely from RAM off a USB stick. The stick holds the OS and your saved configuration; your data lives on ordinary mounted drives. It is built for power users who want a small, transparent, hands-on system — every moving part is a readable shell script you can inspect (and, in a pinch, patch) on the box.

  • The OS is rebuilt in RAM at every boot from packages on the stick plus a small config overlay. A reboot is a factory-fresh OS with your config applied.
  • Nothing persists until you commit. nas commit saves your /etc changes back to the stick. This is a feature: experiments are free, and a reboot always recovers a clean system.
  • Storage is plain Linux. Disks are ordinary filesystems declared in /etc/fstab — no proprietary volume format. Any Linux machine can read your data disks.
  • Data services are supervised. Docker, Samba and NFS are held until the data disk is confirmed mounted, so a missing disk can never quietly fill RAM.
  • CLI-first, appliance discipline. One tool (nas) drives everything; optional extras (web dashboard, notifications, VPNs, UPS) are off until you enable them and safe to ignore.

How it works ↑

Boot sequence

  1. Firmware boots the USB stick (BIOS or UEFI); the kernel + initramfs load.
  2. Your saved config overlay (from the MNASCFG partition) is applied.
  3. The base system + any packages you added install into a RAM filesystem from the on-stick package snapshot and /cfg/cache — no network needed.
  4. The mountnas supervisor mounts your fstab disks, then starts Docker/Samba/NFS once /mnt/nasdata is up.

What lives where

PlaceWhatSurvives reboot?
RAM (/)The running OS, all live changesNo — until nas commit
USB BOOTKernel, packages, boot loadersYes (replaced by nas upgrade)
USB MNASCFG (/cfg)Committed config overlays, package cache, operations logYes
Data disks (/mnt/*)Your files, Docker data, appdata, backupsYes — never touched by commits or upgrades
The one rule that matters: changes live in RAM until nas commit. That includes passwords, fstab edits, installed packages, and Samba users. nas status and the shell prompt track the unsaved count for you.

First boot ↑

  1. Write the release image to a USB stick (4 GB+) with Etcher or dd, boot from it.
  2. Log in as root (no password). The setup wizard starts automatically: hostname, root password, timezone, network — then it saves.
  3. Add your disks (next section), then nas commit.

Finding a headless box

  • The console login screen shows the IP and hostname.local before login — a monitor is enough.
  • mDNS is on: ssh root@mountnas.local works on most LANs.
  • Pre-seed an SSH key: after flashing, drop a file named authorized_keys onto the FAT BOOT partition — it is installed at every boot.
  • The fresh image permits passwordless root SSH so you can get in; the wizard disables that automatically once a password is set.

Choosing hardware

  • RAM is the resource that matters: the whole OS unpacks into RAM at boot. 4 GB is the working minimum; add headroom for your Docker stack (8 GB is comfortable for a typical media box). More disks cost almost no RAM.
  • USB stick: a name-brand stick (A1/high-endurance class helps). It is only read at boot and written on nas commit/upgrade, so wear is low — the real enemy is counterfeit flash: verify a new stick with the baked-in f3 (f3probe /dev/sdX) before trusting it.
  • Disk controllers: motherboard SATA ports are ideal. For many disks, a used LSI HBA flashed to IT mode is the community standard. USB disk enclosures work but are the least reliable path (bridge resets look like disk loss — the watcher will tell you).
  • Network: Intel NICs have the least-troublesome firmware/drivers. 2.5GbE is well supported.
  • CPU: anything x86_64 boots. For media transcoding in Docker, an Intel CPU with Quick Sync (or a cheap Arc/AMD GPU — firmware is baked in) goes a long way.

Storage ↑

Disks are managed in /etc/fstab — like any Linux box. Mount everything under /mnt. One mountpoint is special: /mnt/nasdata must exist to hold Docker data, appdata and backups; Docker/Samba/NFS stay off until it is mounted.

Add a disk

nas disks                         # find the disk; copy its paste-ready fstab line
mkfs.ext4 -L nasdata /dev/sdX1    # blank disk? format it first
vi /etc/fstab                     # e.g.: UUID=<uuid>  /mnt/nasdata  ext4  rw,noatime,nofail  0 2
nas status                        # pre-flight check of your fstab
nas restart                       # mount + start services, no reboot
nas commit                        # persist it
  • Always include nofail — a missing disk must never hang the boot.
  • Mountpoints are created for you from fstab at each boot — no mkdir needed (and do not use x-mount.mkdir).
  • Network filesystems (NFS/CIFS) are never mounted at boot and are refused as /mnt/nasdata — a dead remote must not stall the box. Mount them manually or use them inside Docker.
  • nasdata on a shared disk? Use a bind mount: /mnt/disk1/nasdata  /mnt/nasdata  none  bind,nofail  0 0 (real disk listed first).

Pool disks into one mount (mergerfs)

/mnt/disk1:/mnt/disk2  /mnt/pool  fuse.mergerfs  nofail,allow_other,use_ino,category.create=mfs  0 0

Add after the member disks in fstab; nas commit and reboot (the pool is mounted by the boot path). Keep /mnt/nasdata out of the pool.

Parity (SnapRAID)

  1. Mount data + parity disks in fstab like any other disk.
  2. Configure /etc/snapraid.conf (a commented template ships). Keep /mnt/nasdata out of the array.
  3. Schedule sync/scrub and pipe the result into a notification:
crontab -e
0 3 * * *  snapraid sync 2>&1 | nas notify "snapraid sync"
0 5 * * 0  snapraid scrub -p 8 -o 20 2>&1 | nas notify "snapraid scrub"

SnapRAID gives parity (recover a failed disk), mergerfs gives one namespace — they compose well. Then nas commit.

Parity hygiene: sync vs scrub vs check

  • snapraid sync updates parity to match today's data. Files changed since the last sync are not protected — run it nightly, and after any big copy.
  • snapraid scrub re-reads old blocks and verifies them against parity — this is what catches silent bit rot. -p 8 -o 20 scrubs 8% of the array per run, oldest blocks first: weekly, the whole array gets verified roughly every 3 months.
  • snapraid diff previews what a sync would do (great before syncing after a mass delete — parity still holds the old data until you sync).
  • snapraid check / fix are the recovery tools — see When a disk fails.
  • Avoid syncing while files are being written (pause heavy Docker writers or schedule for quiet hours); the dashboard's Protection card shows how stale parity is.

Disk spin-down & power

MountNAS is careful never to wake a sleeping disk (status checks, temps, SMART and the dashboard are all standby-safe). Spin-down itself is your call — set it per disk with hdparm and persist it as a boot script:

mkdir -p /etc/local.d
cat > /etc/local.d/spindown.start <<'SH'
#!/bin/sh
hdparm -S 241 /dev/disk/by-id/ata-WDC_...   # 241 = 30 min idle; 0 = never
SH
chmod +x /etc/local.d/spindown.start
nas commit

(/etc/local.d is saved by nas commit like the rest of /etc — no lbu include needed; only /etc/init.d has that quirk.) Use /dev/disk/by-id/ paths (stable across reboots). Check a disk's state anytime with hdparm -C /dev/sdX. What keeps disks awake: Docker/appdata living on them (keep that on /mnt/nasdata), nas logs --persist on (targets nasdata), open Samba/NFS handles, and mergerfs create policies that touch every branch. The nasdata system disk is expected to stay awake — spin down the media disks, not that one.

Sharing & services ↑

Samba (Windows/macOS shares)

vi /etc/samba/smb.conf            # add a [share] block (template included)
adduser -D alice                  # system user (if new)
smbpasswd -a alice                # samba password
rc-service samba restart
nas commit

NFS

vi /etc/exports
rc-service nfs restart && nas commit

Docker

Runs with data-root = /mnt/nasdata/docker. Keep compose files and appdata under /mnt/nasdata so they survive a dead USB and travel with the data. docker compose is included.

Remote access (all off by default)

  • Tailscale: rc-update add tailscale default && rc-service tailscale start && tailscale up && nas commit
  • ZeroTier: rc-update add zerotier-one default && rc-service zerotier-one start && zerotier-cli join <net> && nas commit
  • WireGuard: plain kernel WireGuard, wg/wg-quick included.

UPS (NUT) — full walkthrough

NUT is baked in, off by default. A power cut discards uncommitted changes, so a UPS + clean shutdown is the natural completion of the commit model. Most consumer UPSes (APC, CyberPower, Eaton) speak usbhid-ups over USB:

  1. Plug in the UPS data cable and confirm the box sees it: nut-scanner -U (it prints a ready-made config block).
  2. /etc/nut/nut.conf: MODE=standalone
  3. /etc/nut/ups.conf — paste the scanner's block, or the typical: 
    [ups]
  driver = usbhid-ups
  port = auto
  desc = "MountNAS UPS"
  4. /etc/nut/upsd.conf: LISTEN 127.0.0.1
  5. /etc/nut/upsd.users: 
    [monuser]
  password = pickone
  upsmon primary
  6. /etc/nut/upsmon.conf: 
    MONITOR ups@localhost 1 monuser pickone primary
SHUTDOWNCMD "/sbin/poweroff"
  7. Enable and verify: 
    rc-update add nut-upsd default && rc-update add nut-upsmon default
rc-service nut-upsd start && rc-service nut-upsmon start
upsc ups            # shows status, charge, runtime
nas commit

The dashboard's Services card shows charge/runtime once it's running. Dry-run the shutdown path with upsmon -c fsd only when you mean it — it powers the box off.

Admin users

adduser bob wheel && nas commit    # wheel = doas (sudo) rights

Disabling unused services ↑

Everything can be turned off permanently — less RAM, fewer listeners, smaller attack surface. Two mechanisms, depending on how the service is started:

Docker / Samba / NFS (supervisor-managed)

These live in no runlevel — the mountnas supervisor starts them once the data disk mounts — so rc-update del does nothing for them. Set DATA_SERVICES in /etc/conf.d/mountnas to only what you keep:

rc-service docker stop                                     # stop it now
echo 'DATA_SERVICES="samba nfs"' > /etc/conf.d/mountnas    # a box that does not use Docker
nas status                                                 # verify: docker now listed as disabled
nas commit                                                 # REQUIRED — the setting is RAM-only until committed

nas status and this dashboard understand the override: a deliberately disabled service is listed as disabled, never warned about. Re-enable by editing the list back (or deleting the file), then nas restart and nas commit again.

Runlevel services

rc-service <name> stop && rc-update del <name> default && nas commit
ServiceSafe to disable if…You lose
avahi-daemonyou use IPs/DNS, not .localmDNS discovery (stop dbus too if nothing else needs it)
chronydnot recommendedtime sync — drift breaks SnapRAID timestamps, TLS, logs
crond⚠️ think twicescheduled jobs including the 15-min disk-loss watcher and SnapRAID syncs

rpcbindyou don't serve NFSnothing else uses it
acpidheadless, never power-buttonedclean shutdown on the power button
sshd⚠️ console-only administrationall remote access — have a monitor/keyboard first

Optional services

Tailscale, ZeroTier and NUT ship off; if you enabled one, the same stop + rc-update del + commit removes it. The dashboard: nas web off && nas commit. WireGuard has no daemon — wg-quick down <iface> and remove your local.d/cron hook.

Don't apk del baked-in packages to disable a service — nas upgrade's world reconciliation restores the base package set, so the binaries return by design. Disabling the service is the supported, upgrade-proof way; idle binaries cost nothing (the OS lives in RAM regardless).

Monitoring & alerts ↑

nas status — the pre-flight check

nas status validates health and storage config without waking sleeping disks; exit code 1 on any failure makes it cron/monitoring friendly. --deep adds SMART/SnapRAID/time-sync; --json is machine-readable for Uptime Kuma, Zabbix, Homepage, etc.

This web dashboard

Enable/disable with nas web on [port] / nas web off (default port 8080). Then nas commit — like every setting, on/off lives in RAM until committed, and uncommitted the dashboard is gone (or back) after a reboot; nas web status tells you when the state is unsaved. It is deliberately read-only: a root-run job renders a static page into RAM every 2 minutes and a tiny server (running as nobody) serves it — no request-time code, no forms, nothing to attack, no disk writes. Manage the box over SSH. Everything is on the one page: health, services, disks, hardware, your added packages, and a collapsible tail of the last 100 syslog lines; /status.json serves the data machine-readable.

Notifications

List sinks in /etc/mountnas/notify.conf, one per line; every alert goes to all of them:

ntfy:https://ntfy.sh/your-secret-topic          # push to your phone, no relay needed
email:you@example.com                           # needs /etc/msmtprc (msmtp) configured
webhook:https://example.com/hook                # generic JSON POST
slack:https://hooks.slack.com/services/...      # also Mattermost-compatible
discord:https://discord.com/api/webhooks/...
gotify:https://gotify.example.com/message?token=...

Test with nas notify --test, persist with nas commit. What alerts automatically: data-disk loss (disconnect, dead mount, filesystem gone read-only — once, on the transition), failed upgrades, and anything you pipe in yourself (... | nas notify "subject"). Sink docs: ntfy, msmtp (email), gotify.

SMART disk health (smartmontools)

smartd runs out of the box and never wakes spun-down disks. For failure alerts, add ONE line to /etc/smartd.conf and restart smartd:

DEVICESCAN -n standby,q -m root -M exec /usr/libexec/mountnas/smartd-notify   # via your sinks
DEVICESCAN -n standby,q -m you@example.com                                    # or classic email

Health digest (optional)

crontab -e
0 8 * * 1  /usr/libexec/mountnas/health-digest    # Monday-morning summary via your sinks

Operations log

nas history shows every setup, commit (with note), rollback, backup, upgrade and shutdown — timestamp, who, outcome. Zero setup; persists without a commit.

Persistent system logs (optional)

nas logs --persist on moves syslog to /mnt/nasdata/logs (rotated automatically) so a crash leaves history behind — tradeoff: periodic writes keep that disk awake.

When a disk fails ↑

Don't panic: the OS runs from RAM, so a failed data disk never takes the system down — the box stays up, holds Docker/Samba/NFS off the broken storage, and tells you. You'll hear about it three ways: a push through your notification sinks (once, on the transition), a red [FAIL] in nas status, and the dashboard turning red.

What the state means, and what to do

StateIt meansDo this
disconnectedThe device itself is gone (died, yanked, USB reset)Check power/cabling/enclosure. Reattach, then nas restart. If the disk is truly dead → replacement drill below.
mountfailDevice present but the filesystem won't mount (or the mount went dead)dmesg | tail for the reason. Unmounted repair: e2fsck -f /dev/sdX1 (or xfs_repair), then nas restart. Repeated failures = a dying disk — replace it.
read-onlyThe filesystem hit errors and protected itself (errors=remount-ro)Treat as pre-failure. Copy anything irreplaceable off it now, then fsck offline; SMART (smartctl -a) usually names the culprit. Plan a replacement.

Replace a failed data disk (SnapRAID)

This is the drill parity exists for — the array can rebuild the lost disk's contents:

1  stop writers:  docker stop $(docker ps -q)         # keep the array quiet
2  swap hardware; format the new disk:  mkfs.ext4 -L disk1 /dev/sdX1
3  /etc/fstab: point the SAME mountpoint (/mnt/disk1) at the new disk's UUID/label
4  nas restart                                        # new empty disk mounts
5  snapraid fix -d d1 -l /root/fix.log                # rebuild d1 from parity (hours)
6  snapraid check -d d1                               # verify the rebuild
7  snapraid sync && nas commit

d1 is the data name from /etc/snapraid.conf, not the device. Anything written after your last sync is not in parity — that gap is why the nightly sync matters.

Replace or grow the parity disk

Parity holds no user data — swap it freely: mount the new (equal-or-larger) disk at the same /mnt/parity1 path, then snapraid sync rebuilds parity from scratch. Growing a data disk works the same way: parity disks must always be at least as large as the largest data disk.

Add a disk to the array

mkfs + fstab + nas restart                 # like any disk (Storage section)
echo "data d3 /mnt/disk3" >> /etc/snapraid.conf
snapraid sync && nas commit

Maintenance ↑

Committing & rolling back config

nas changes --diff                 # review what would be saved
nas commit -m "added media share"  # save, with a label
nas rollback --list                # previous saved configs (with your labels)
nas rollback 1 && nas reboot       # restore one (applies at next boot)

Backup (the rollback net)

nas backup images the whole boot USB (OS + saved config) to a verified gzip file — default /mnt/nasdata/backups. Copy it off the box. Recovery = write it to a different stick and boot. Data disks are separate — back those up with restic/rclone (both included).

Off-box data backup (restic)

restic gives encrypted, deduplicated, versioned backups to SFTP, S3/B2, or any rclone remote. One-time setup — keep the credentials in /root (it persists via lbu):

cat > /root/.restic.env <<'ENV'
export RESTIC_REPOSITORY=sftp:backup@othernas:/backups/mountnas
export RESTIC_PASSWORD='a-long-passphrase-you-saved-somewhere-safe'
ENV
chmod 600 /root/.restic.env
. /root/.restic.env && restic init          # once per repository

For S3/B2 instead: RESTIC_REPOSITORY=s3:s3.amazonaws.com/bucket plus AWS_ACCESS_KEY_ID/AWS_SECRET_ACCESS_KEY lines in the same file. Then schedule it, piped into your alert sinks:

crontab -e
30 2 * * *  . /root/.restic.env && restic backup /mnt/disk1/photos /mnt/nasdata/appdata 2>&1 | nas notify "restic backup"
0  6 * * 0  . /root/.restic.env && restic forget --keep-daily 7 --keep-weekly 4 --keep-monthly 6 --prune 2>&1 | nas notify "restic prune"

Then nas commit (saves the env file and the crontab). Restore anywhere with restic restore latest --target /tmp/r — test that once before you need it. Remember: restic protects your data; nas backup protects the boot stick; SnapRAID protects against disk loss. The three cover different failures.

Upgrading

nas upgrade --check                                  # is a newer release out?
nas backup                                           # required first — no automatic rollback
nas upgrade https://github.com/.../mountnas-X.img.gz # or a local file; then reboot

Upgrades rewrite the OS in place; your config and data are untouched, added packages are preserved. See UPGRADE.md in the repo for details and recovery.

Installing packages

apk add <pkg> works out of the box (repos are pinned to this release). After installing: nas commit — packages are cached on the stick and reinstall at every boot, even offline. Missing device firmware: find the file with dmesg | grep -i firmware, apk add linux-firmware-<vendor>, commit, reboot.

Recovery

  • Dead USB stick: write your latest nas backup image to a new stick and boot — OS and config restored exactly. No backup? Flash a fresh release and reconfigure; data disks are untouched.
  • Lost root password: mount MNASCFG on another Linux box, unpack the overlay, blank the root hash in etc/shadow, repack, boot, passwd, nas commit. (Full steps in the README.)
Never re-flash the release image over a running box (it erases the saved config — the image is for first install; upgrades go through nas upgrade), and never boot with two MountNAS sticks attached (their disk labels collide).
